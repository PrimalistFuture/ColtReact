Server-Side Routing

Traditional routing is server-side. Clicking a link causes browser to request a new page and replace the entire DOM. The Server decides what HTML to return based on URL requested and the entire page refreshes. The information from the old page is totally gone and no trace of it will be on the new page.

Fake Client-Side Routing

Client-side routing handles mapping between URL bar and the content a user sees via browser rather than via server.

Client-side routing essentially fakes traditional server-side routing. All of your data exists within the page already, but browser will only show what you want it to. When a link is clicked it won't take you to a new page, but a it will refresh with the new content.

Issues: The browser doesn't really know that things have changed, and the forward and back buttons don't work. We don't get URLs that would let a user navigate directly do a page. And the user can't bookmark specific pages.

Sites that exclusively use client-side routing are called single-page applications. They use JavaScript to manipulate the URL bar with a Web API called History

It works, but its just okay.

React Router and real client-side routing

There are many tools you can use to make client-side routing, and React Router is one such tool. It is not built into React, or really even an offical add-on. But it is very popular.

To install, npm install react-router-dom. It must be imported at the top, import {BrowserRouter} from "react-router-dom";. There are other types of routers you can look into, but this one is fine.
